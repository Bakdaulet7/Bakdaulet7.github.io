Row-Level Security in Power BI: A Practical Guide

Power BI
Security
Everything you need to know about implementing RLS in Power BI — static roles, dynamic RLS, and embedding with multi-tenant access control.
Published

November 8, 2025

Row-Level Security (RLS) is one of those features that looks simple until you’re implementing it for a real production deployment. Here’s a practical guide based on what I’ve built at Umojo.

The two flavors: Static vs. Dynamic

Static RLS means you hardcode filter values into a role. Useful for simple, well-defined groups:

[Region] = "North America"

This works great when your user groups are stable and small. When you have 200 tenant-specific views, it breaks down immediately.

Dynamic RLS uses DAX functions to make filtering data-driven, keying off the currently authenticated user:

[Email] = USERPRINCIPALNAME()

This is what most production systems need. One role, infinite tenants.

The security table pattern

The pattern I use most often: a dedicated security table that maps users to their accessible data.

SecurityTable
├── UserEmail (text)
├── RegionID (int)
└── ClientID (int)

Your RLS DAX then becomes:

[ClientID] IN
  CALCULATETABLE(
    VALUES(SecurityTable[ClientID]),
    SecurityTable[UserEmail] = USERPRINCIPALNAME()
  )

This approach decouples access control from your data model, making it easy to add/remove access without touching DAX or Power BI Desktop.

Embedding with RLS

When embedding reports via the Power BI Embedded API, you set the effective identity at token generation time. This is where multi-tenant dashboards get interesting — you can scope a single embedded report to an arbitrary user identity without that user even having a Power BI license.

const embedToken = await powerbiClient.reports.generateToken(
  workspaceId,
  reportId,
  {
    accessLevel: 'View',
    identities: [{
      username: tenantUserEmail,
      roles: ['TenantRole'],
      datasets: [datasetId]
    }]
  }
);

This is the pattern we use at Umojo to serve curb management data to city clients — one report, RLS-scoped per city.

Common pitfalls

1. Testing with your own account: You’ll always see everything unless you explicitly test with “View as role” in Power BI Desktop. Build a test matrix and use it religiously.

2. Forgetting bidirectional cross-filter: If your security table has a relationship to a bridge table, make sure cross-filter direction is set correctly, or users will see data they shouldn’t.

3. USERPRINCIPALNAME vs. USERNAME: In cloud, UPN is the email. In on-premises gateway scenarios, USERNAME returns the Windows identity. Know your environment.

4. Service Principal embedding: When using service principals for embedding, the identities array must include a dataset reference or the token generation will fail silently.

Validating RLS in production

Don’t trust your staging environment alone. I always build a small validation report that shows each user exactly which rows they’re seeing, and cross-reference against the security table. Surfacing this to the right ops person early catches misconfigured access before clients notice.
