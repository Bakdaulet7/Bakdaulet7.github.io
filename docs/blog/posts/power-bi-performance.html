Optimizing Power BI Report Load Times

Power BI
Performance
Practical techniques to reduce Power BI report load times — from DAX optimization and model design to storage mode decisions.
Published

March 19, 2024

Slow Power BI reports kill adoption. I’ve worked on dashboards that took 45 seconds to load and ones that loaded in under 2. Here’s what makes the difference.

Diagnose before you optimize

Use Performance Analyzer (View → Performance Analyzer) to record exactly how long each visual takes to render and which DAX queries are running. Never guess — measure.

For deeper investigation, connect DAX Studio to your report and use Server Timings to see storage engine vs. formula engine time. If most of your time is in the formula engine, your DAX needs work. If it’s storage engine, your model structure needs work.

DAX optimizations

Avoid iterator functions over large tables

SUMX, AVERAGEX, and friends iterate row by row. Over millions of rows, this is slow.

-- Slow
SlowMeasure = SUMX(Orders, Orders[Quantity] * Orders[UnitPrice])

-- Faster: pre-compute in a calculated column, or restructure
FastMeasure = SUMX(Orders, Orders[ExtendedPrice])

If the multiplication is always the same, make it a calculated column so it runs once at refresh time, not at query time.

Replace CALCULATE with simpler alternatives

Every CALCULATE modifies the filter context — it’s powerful but not free. For simple filtered aggregations, sometimes a relationship and a base measure is faster.

Use variables

Variables cache the result of an expression so it isn’t evaluated multiple times in the same DAX query:

SalesMeasure =
VAR TotalSales = SUM(Orders[Amount])
VAR Target = [SalesTarget]
RETURN
    DIVIDE(TotalSales - Target, Target)

Without VAR, SUM(Orders[Amount]) might be evaluated multiple times.

Model design

Use star schema, not flat tables

A fact table surrounded by dimension tables lets the VertiPaq engine compress data far more efficiently than a single wide flat table. If your model is one giant table with 80 columns, restructure it.

Remove unused columns

Every column you import lives in memory. If you’re not using CustomerMiddleName in any visual or DAX, drop it in Power Query before loading. This alone can halve model size on data-rich tables.

Choose the right storage mode

Scenario Storage Mode
Data fits in memory, fast queries needed Import
Huge data, acceptable query latency DirectQuery
Mix of large + small tables Composite
Fabric Lakehouse, near-real-time Direct Lake

DirectQuery reports are almost always slower than Import — only use it when data volumes or freshness requirements force your hand.

Visuals and report design

  • Reduce visuals per page. Each visual triggers at least one DAX query. A report with 25 visuals on one page fires 25+ queries on load.
  • Use Bookmarks to show/hide visuals rather than putting everything on the canvas at once.
  • Avoid high-cardinality slicers. A slicer on a column with 50,000 unique values is expensive to render.
  • Disable “show items with no data” on visuals unless you need it — it forces additional queries.

The biggest win I’ve seen

The single biggest improvement I’ve ever gotten: removing unused relationships from the data model. Invisible bidirectional relationships that “seemed harmless” were forcing the engine to walk the full graph on every query. Removing two unused many-to-many relationships cut a dashboard from 22 seconds to 4 seconds.

Always audit your relationships after a model has been in production for a while. Relationships accumulate. Not all of them should stay.
